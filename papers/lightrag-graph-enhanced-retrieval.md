**Source:** HKUDS — [github.com/hkuds/lightrag](https://github.com/hkuds/lightrag)

**Published at:** EMNLP 2025  
**Authors:** Zirui Guo, Lianghao Xia, Yanhua Yu, Tu Ao, Chao Huang (Hong Kong University)

---

## Why We're Reading This

LightRAG sits at the intersection of two things we care about: knowledge graph construction and retrieval for agents. Where Graphiti is a memory engine (data flows in continuously, agents query it in real time), LightRAG is a retrieval system (documents are indexed once, queries retrieve from them). Different use case — but the graph construction approach and the dual-level retrieval architecture are directly relevant to how we think about what agents need from a memory layer at query time.

The specific insight worth taking: not all queries are the same. Some want a specific fact. Others want a broad theme. A retrieval system that treats every query the same — top-k vector similarity — handles neither well. LightRAG's solution to this is the dual-level retrieval pattern.

---

## 1. The Problem With Standard RAG

Standard RAG embeds document chunks into vectors and retrieves the top-k most similar chunks at query time. This works for simple fact lookup. It breaks in two ways:

**The specificity problem:** "What did the client say about budget in our last meeting?" → a single, specific fact buried in a long transcript. Vector similarity retrieves the whole chunk, not the specific sentence. The answer is drowned in noise.

**The comprehensiveness problem:** "What are the recurring themes across all our sales calls this quarter?" → no single chunk contains this. Vector similarity retrieves individual chunks, each with local context only. The cross-document, thematic answer is invisible.

These two failure modes pull in opposite directions. Fixing one typically makes the other worse. LightRAG solves both by running two different retrieval strategies and combining their results.

---

## 2. How LightRAG Works

### Step 1: Graph Construction (at indexing time)

When documents are ingested, LightRAG does three things:

**Entity and relationship extraction** — the LLM reads each text chunk and extracts: entities (people, companies, concepts, events) and the relationships between them as typed edges. Same bottom-up approach as Graphiti — structure is discovered from raw text, not predefined.

**LLM profiling** — for each extracted entity and relationship, a key-value pair is created: the key is a normalized string used for retrieval, the value is a summarized description generated by the LLM. This is important — the entity isn't just a string, it's a summary of what's known about it.

**Deduplication** — identical entities and relationships extracted from different document chunks are merged into single nodes/edges. `"OpenAI"` appearing in 50 different chunks becomes one node with one consolidated summary. This is LightRAG's entity resolution step: LLM-based, string-matching with summarization merge.

The result: a knowledge graph where nodes are named entities with LLM-generated summaries and edges are typed relationships with their own summaries.

### Step 2: Dual-Level Retrieval (at query time)

When a query arrives, LightRAG runs two parallel retrieval strategies:

**Low-level retrieval** — targets specific entities and their direct relationships.
- Extracts local keywords from the query (specific named entities, precise terms)
- Matches those keywords against entity nodes in the graph
- Retrieves the matched entities plus their immediate neighbors
- Best for: specific, factual queries with a clear subject

**High-level retrieval** — targets broader themes and cross-entity patterns.
- Extracts global keywords from the query (concepts, themes, abstract terms)
- Matches against relationship edges and entity summaries, not just node names
- Retrieves across multiple entities and relationships that share the theme
- Best for: conceptual queries that span many entities

Both run simultaneously. Results are merged and passed to the LLM for final answer generation.

### The Three-Step Query Flow

```
Query arrives
  ↓
LLM extracts: local keywords + global keywords simultaneously
  ↓
Low-level: vector match against entity nodes → retrieve node + neighbors
High-level: vector match against relation edges → retrieve cross-entity context
  ↓
Merge results → LLM generates answer from combined context
```

## 3. Performance vs. Competitors

**vs. GraphRAG:**
- LightRAG wins on larger datasets (5M+ tokens), especially on diversity and breadth metrics
- Retrieval cost: LightRAG uses <100 tokens and 1 API call per query. GraphRAG uses 610,000 tokens and hundreds of API calls. The cost difference is ~6,000x.
- Why: GraphRAG generates community summaries at indexing time (expensive) and queries them at runtime (expensive). LightRAG pushes work to indexing, keeps queries cheap.

**vs. Naive RAG:**
- LightRAG wins 65–83% of queries across datasets
- The gap widens significantly on complex, cross-document queries where chunk-based retrieval can't see interdependencies
- On simple single-chunk factual queries, the gap narrows — naive RAG handles those adequately

**The tradeoff:** LightRAG's indexing is more expensive than naive RAG (requires LLM extraction at ingest time). Query time is comparable. The investment is made once at indexing; retrieval is fast thereafter.

---

## 4. Incremental Updates

LightRAG supports incremental ingestion — new documents can be added without reprocessing the entire corpus. New entities and relationships are extracted and merged into the existing graph. This is the same principle as Graphiti's incremental update model: the graph evolves as data arrives, rather than requiring batch recomputation.

The deduplication step handles merging: if a new document mentions an entity already in the graph, LLM profiling updates the entity's summary to incorporate the new information. Old summaries aren't deleted — they inform the updated one.

---

## 5. Where LightRAG Fits in the Retrieval Landscape

Three systems we've now studied, each solving a different retrieval problem:

| System | Optimized for | Graph type | Updates | Entity resolution |
|---|---|---|---|---|
| **Naive RAG** | Fast, simple fact lookup | None | Easy — just re-embed | None |
| **GraphRAG** | Thematic, community-level queries | Pre-computed communities | Expensive — full recomputation | LLM-based at indexing |
| **LightRAG** | Both specific and thematic queries | Entity-relationship graph | Incremental | LLM-based + deduplication |
| **Graphiti** | Real-time agent memory with history | Temporal fact graph | Real-time, streaming | LLM name matching |

LightRAG is best understood as: GraphRAG's retrieval quality at a fraction of the query cost, with incremental update support added. It's still a retrieval system, not a memory system — it doesn't track temporal validity of facts, doesn't model decay, and isn't designed for real-time signal ingestion.

## 6. The Dual-Level Pattern and GTM Memory

The dual-level retrieval pattern is worth internalizing regardless of whether you use LightRAG directly. The underlying insight applies to any memory query system:

**Some agent questions are low-level (specific):**
- "What did John at Acme say about budget in the last meeting?"
- "When did this contact last visit the pricing page?"
- "What is the current pipeline stage for this contact?"

These need precise, entity-specific retrieval. Fast, low-noise, exact or near-exact match.

**Some agent questions are high-level (thematic):**
- "What are the most common objections across all evaluating-stage contacts this month?"
- "Which companies in our pipeline have shown pricing interest in the last 30 days?"
- "What patterns do we see across contacts who converted to client?"

These need cross-entity, relationship-aware retrieval. Chunk-based vector search can't answer them — the answer lives in the aggregate, not in any single record.

Proply's current architecture handles low-level queries well: structured lookups by contact ID, stage, score, recency. The high-level pattern — aggregate queries across the full pipeline — is where graph-style retrieval adds value. Whether that's implemented with LightRAG, a dedicated analytics layer, or Postgres aggregates depends on the query patterns. The dual-level framing is the useful thing.

---

## 7. What We're Taking From This

**Graph construction at indexing time pays off at query time.** LightRAG's 6,000x query cost advantage over GraphRAG comes from doing the expensive LLM work upfront. The same principle applies to Proply's stage compute: doing the stage calculation on every activity write (expensive trigger) makes every query instantaneous (cheap read). Pay at write time, not at query time.

**Deduplication as a first-class step.** LightRAG explicitly merges identical entities across chunks before they reach the graph. This is the right order of operations — resolve before storing, not after. Proply's waterfall does the same: resolve contact identity before logging any activity.

**The dual-level framing for memory queries.** Agents querying Proply need both: specific contact context (low-level) and pipeline-wide patterns (high-level). The retrieval strategy should match the query type. Structured DB queries handle low-level; aggregate queries or a graph layer handle high-level.

**Incremental > batch.** LightRAG's incremental update support is the same conclusion Graphiti reaches: memory systems for agents can't wait for batch processing windows. Every write should update the graph immediately.

---

## Key Takeaway

LightRAG's contribution is the dual-level retrieval pattern: run low-level (specific entity) and high-level (thematic, cross-entity) retrieval simultaneously and merge. Neither naive RAG nor GraphRAG does both well. The underlying principle — that queries have different granularities and retrieval strategies should match them — is architecture-independent and worth applying anywhere agents need to query memory at different levels of abstraction.
