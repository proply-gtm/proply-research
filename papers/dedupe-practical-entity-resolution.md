## Why We're Reading This

Dedupe is the most widely used open-source entity resolution library. Where the academic survey gives you the theory, Dedupe shows you what a practical, production-grade ER system actually looks like in code. The interesting part isn't just what it does — it's *how* it solves the labeling problem: instead of requiring thousands of hand-labeled training pairs, it uses active learning to figure out which examples a human actually needs to label to produce a good model.

---

## 1. What Dedupe Does

Dedupe is a Python library for fuzzy matching, deduplication, and record linkage on structured data. It answers one question: *given a dataset of records with messy, inconsistent string fields, which records refer to the same real-world entity?*

It handles three distinct tasks:
- **Deduplication** — finding duplicate records within a single dataset
- **Record linkage** — matching records across two datasets
- **Gazetteer matching** — matching records against a canonical reference dataset

For GTM context: deduplication = cleaning your own contact database. Record linkage = matching incoming webhook contacts against existing contacts. Gazetteer = matching against a known company database.

---

## 2. The Core Approach

### Step 1: Blocking (Indexing)

Before comparing anything, Dedupe reduces the comparison space using **predicates** — blocking keys derived from field values. Instead of comparing every record to every other record (O(n²)), it generates candidate pairs that share at least one blocking predicate.

Examples of predicates Dedupe learns automatically:
- First 4 characters of `name` field
- Whole token from `company` field
- Metaphone encoding of `last_name`
- Exact match on `email_domain`

The key insight: **Dedupe learns which predicates to use from your data and labels.** It doesn't require you to specify them — it searches the space of possible predicates and selects the combination that covers the most likely matches while generating the fewest false comparisons.

### Step 2: Active Learning (The Labeling Engine)

This is what makes Dedupe different from most ER approaches.

Training a matcher normally requires thousands of labeled pairs (match / no match). Dedupe cuts this down to dozens using active learning:

1. Generate a large pool of candidate pairs
2. Identify the **most uncertain** pairs — the ones where the current model is least confident
3. Present those specific pairs to a human for labeling
4. Retrain the model on the new labels
5. Recalculate uncertainty across all pairs
6. Repeat until confidence is high enough

The insight: you don't need to label representative examples. You need to label the examples that will change the model's behavior the most. A pair the model is 99% confident about teaches nothing. A pair it's 51% confident about is maximally informative.

In practice, 10-30 labeled pairs often produce a model that generalizes well to millions of records.

### Step 3: Matching

Once trained, Dedupe calculates a match probability for every candidate pair using a logistic regression model over the feature vectors generated by comparing field values.

Field comparison methods available:
- `String` — edit distance for free-form text
- `ShortString` — optimized for short strings like names
- `Text` — TF-IDF cosine similarity for longer text
- `Exact` — binary exact match
- `DateTime` — date comparison with configurable tolerance
- `LatLong` — geographic proximity

Each field gets a learned weight — the model figures out that `email` is more discriminative than `first_name`, for example.

### Step 4: Clustering

Matched pairs are grouped into clusters using **hierarchical clustering with centroid linkage**. Each cluster = one canonical entity. The similarity threshold that determines cluster membership is tunable — lower threshold = more aggressive merging.

## 3. What Dedupe Gets Right

**The labeling bottleneck is the real problem.** Most entity resolution systems fail in practice not because the algorithm is wrong, but because you can't get enough labeled training data. Dedupe's active learning approach directly attacks this — it makes every label count.

**Blocking is learned, not hardcoded.** Traditional ER requires engineers to manually define blocking rules. Dedupe treats blocking key selection as a search problem and solves it automatically. This matters enormously in practice — poorly chosen blocking keys either miss too many true matches or generate too many comparison pairs to be practical.

**Field weights are learned.** You don't have to know whether `email` or `company_name` is more discriminative for your dataset. The model figures it out from labeled examples.

---

## 4. What Dedupe Doesn't Handle Well

**Streaming / incremental updates.** Dedupe is designed for batch processing. When a new record arrives in real time, it can't easily integrate it without re-running large parts of the pipeline. This is the central limitation for webhook-based systems like Proply.

**Unstructured text.** Dedupe requires structured fields. It can't extract entities from raw text — it can only compare records that already have defined fields (`name`, `email`, `company`, etc.).

**Semantic similarity.** Dedupe handles fuzzy string similarity well but doesn't natively handle semantic equivalence — it won't know that `IBM` and `International Business Machines` should match without a custom field comparator.

**Contextual disambiguation.** `Apple (tech)` vs. `Apple (fruit)` — Dedupe has no mechanism for using surrounding context to disambiguate. It only looks at the record's own fields.

## 5. The Active Learning Pattern Applied to GTM

The active learning insight is transferable beyond deduplication. The underlying principle — *surface the most uncertain cases to a human, not the most representative ones* — applies anywhere a system needs to build a labeled dataset efficiently.

For Proply, this pattern is relevant in two places:

**ICP scoring calibration.** Instead of asking a user to label 100 random contacts as ICP fit or not, surface the 10 contacts the model is least certain about. Each label teaches the most. This is exactly how Dedupe's active learning works, applied to a different classification task.

**Identity resolution edge cases.** The hard cases in the waterfall — where the model isn't sure if two records are the same person — are the cases worth surfacing for human review. Rather than auto-merging or auto-creating, present the uncertain pair with a confidence score and let the user decide. That label then improves the model.

## 6. Comparison to Graphiti's Approach

| Dimension | Dedupe | Graphiti |
|---|---|---|
| **Input** | Structured records (defined fields) | Raw text or JSON episodes |
| **Entity extraction** | Not applicable — fields already exist | LLM extracts entities from text |
| **Resolution method** | Machine learning + active learning | LLM name matching + similarity |
| **Blocking** | Learned predicates, automatic | Hybrid search (embeddings + BM25) |
| **Training requirement** | Needs labeled pairs (10-30 with AL) | Zero-shot via LLM |
| **Streaming support** | Weak — batch-oriented | Strong — incremental by design |
| **Semantic matching** | Weak — string similarity only | Strong — LLM handles semantic equivalence |
| **Contextual disambiguation** | None | Limited — depends on LLM context window |

The practical conclusion: Dedupe is better when your data is already structured and you have time to train a model. Graphiti is better when your data is unstructured and you need zero-shot extraction. For Proply — structured webhook payloads with defined fields — Dedupe's approach (learned blocking + field-weighted matching) is architecturally closer to what's needed than Graphiti's LLM extraction.

## 7. What This Changes for Proply

**The current waterfall is Stage 1 of a full ER pipeline.** What Proply does now (external ID → email → LinkedIn → create) is blocking + exact matching. It handles the easy cases well. What it doesn't have is a fuzzy matching stage for cases where no exact match exists but a true match probably does.

**The gap:** `Ben Glinder` from one source + `Bennet Glinder, bennet@proply.com` from another, arriving months apart. No email in the first record, no external ID match. Current behavior: creates two contacts. Correct behavior: merges them.

**The fix, when we need it:** Add a fuzzy name + company matching step before the "create" fallback. Dedupe's approach — compute edit distance on name + exact match on email domain + phonetic encoding — is the right pattern. Below a confidence threshold, flag for human review rather than auto-merging.

**Active learning for ICP.** The most direct application of Dedupe's core innovation to Proply's product: instead of asking users to set up ICP criteria from scratch, surface the 10 most uncertain contacts and ask "ICP fit or not?" Each answer calibrates the scorer. This is the labeling approach Dedupe proved works.

---

## Key Takeaway

Dedupe's lasting contribution isn't the algorithm — it's the insight that **the bottleneck in entity resolution is labeling, not matching.** Active learning solves this by finding the examples that teach the most per label. The field-weighted, learned-blocking approach it takes is the right architecture for structured data. The active learning pattern — surface uncertainty, not representativeness — is transferable to any place in the product where a model needs human signal to improve.
